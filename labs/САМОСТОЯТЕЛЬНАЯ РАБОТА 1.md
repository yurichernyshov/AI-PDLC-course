**2. Центральный вопрос самостоятельной работы**
***Что означает «подготовить инфраструктуру» — и почему это больше, чем просто установить нужные пакеты?***
Установить пакеты — это технически простая задача: одна команда в терминале. Но «подготовить инфраструктуру» в контексте этого курса означает нечто большее: создать **воспроизводимое, документированное рабочее окружение**, которое любой другой участник команды сможет развернуть на своей машине, следуя только письменной инструкции, без обращения к вам за помощью.
Это кажущееся простым требование на самом деле является первым, небольшим упражнением в инженерной дисциплине, которая пронизывает весь курс: **воспроизводимость как ценность**. Система, которую можно запустить только на вашем компьютере, в вашем специфическом окружении, с пятью неочевидными шагами настройки, которые «я помню, но не успел записать» — это не инженерное решение. Это прототип, застрявший в состоянии «работает только у меня». Воспроизводимое окружение — первый шаг из этого состояния.

**5. Задача 1: Настройка локального окружения**
**5.1. Почему нужно изолированное окружение — подробное объяснение**
Если вы уже работали с Python, вы, возможно, сталкивались с ситуацией: устанавливаете новый пакет для одного проекта, и что-то перестаёт работать в другом. Это происходит потому, что разные проекты могут требовать разные версии одних и тех же зависимостей, и если все пакеты установлены «в систему» — они неизбежно конфликтуют.
Изолированное виртуальное окружение решает эту проблему: для каждого проекта создаётся собственная, независимая «папка» с нужными пакетами нужных версий. Проект этого курса существует в своём окружении и не знает о пакетах других ваших проектов — и наоборот.
**Это не просто «хорошая практика» — это обязательное требование данного курса**, потому что репозиторий проекта (создаваемый на ЛР 1.2) должен содержать файл зависимостей, позволяющий любому участнику команды воспроизвести идентичное окружение одной командой.
**5.2. Выбор инструмента управления окружением**
У Python есть несколько инструментов для создания изолированных окружений. Ниже представлены наиболее распространённые с кратким описанием, позволяющим сделать обоснованный выбор. Принцип инфраструктурного документа курса применяется и здесь: вы свободны в выборе инструмента, но выбор должен быть зафиксирован и обоснован.
**Рекомендация для тех, кто не имеет сложившихся предпочтений:** начните с venv — он встроен в Python, не требует дополнительной установки, и его достаточно для всего, что вам понадобится в этом курсе.
**5.3. Пошаговая инструкция: создание окружения с **venv
**Шаг 5.3.1. Проверьте, что Python установлен и доступен из командной строки.**
Откройте терминал (в Windows — командная строка или PowerShell, в macOS/Linux — Terminal) и выполните:
python --version
или, если предыдущая команда не сработала:
python3 --version
Вы должны увидеть вывод вида Python 3.10.x (или более новую версию). Если команда не найдена — установите Python с официального сайта (python.org). Минимально необходимая версия для этого курса — Python 3.10.
**Шаг 5.3.2. Создайте директорию для проекта.**
mkdir ai-dev-course-project
cd ai-dev-course-project
Эта директория станет корнем репозитория проекта — именно туда на ЛР 1.2 вы развернёте git-репозиторий с обязательной структурой директорий.
**Шаг 5.3.3. Создайте виртуальное окружение внутри директории проекта.**
python -m venv .venv
Команда создаст скрытую директорию .venv внутри вашей папки проекта. Она содержит изолированную копию интерпретатора Python и будет хранить все зависимости проекта.
**Важно:** имя .venv начинается с точки — в Linux/macOS это делает папку скрытой при обычном просмотре (ls без флага -a), что удобно. В Windows точка не делает папку скрытой, но соглашение сохраняется для единообразия.
**Шаг 5.3.4. Активируйте виртуальное окружение.**
*В **macOS** / Linux:*
source .venv/bin/activate
*В** Windows (**командная** **строка**):*
.venv\Scripts\activate.bat
*В** Windows (PowerShell):*
.venv\Scripts\Activate.ps1
После активации вы увидите изменение приглашения командной строки — в начале строки появится (.venv) или аналогичная пометка. Это значит, что все команды python и pip теперь относятся к изолированному окружению, а не к системному Python.
**Как понять, что активация прошла успешно.** Выполните:
which python  # macOS/Linux
# или
where python  # Windows
Путь должен указывать внутрь директории .venv вашего проекта, а не на системный Python.
**Шаг 5.3.5. Обновите **pip** до актуальной версии.**
pip install --upgrade pip
Это хорошая практика: старые версии pip иногда некорректно разрешают зависимости или не поддерживают форматы метаданных новых пакетов.
**Шаг 5.3.6. ****Добавьте **.venv** ****в **.gitignore**.**
Виртуальное окружение не должно попадать в git-репозиторий — оно воспроизводится на каждой машине из файла зависимостей, а не хранится в репозитории. Создайте файл .gitignore (или добавьте строку в уже существующий):
.venv/
__pycache__/
*.pyc
.env
.env.attack
Обратите внимание: .env тоже в списке — это защита секретов (подробнее в разделе 6.5).

**6. Задача 2: Получение доступа к API модели**
**6.1. Почему начинать нужно немедленно — ещё раз о сроках**
Получение доступа к API языковой модели — единственный шаг этой самостоятельной работы, который **не зависит полностью от вашего времени и усилий**. Часть времени уйдёт на ожидание: верификацию аккаунта провайдером, проверку документов (если требуется), активацию бесплатного лимита. Разные провайдеры тратят на это от 10 минут до нескольких рабочих дней.
Если вы начнёте этот шаг за день до лабораторной работы 1.1 — вы рискуете прийти на занятие без работающего доступа и провести его, наблюдая за работой товарищей вместо самостоятельной практики.
**6.2. Как выбрать провайдера модели**
Согласно принципу инфраструктурного документа (раздел 4.1), конкретный провайдер — ваш выбор. Ниже перечислены критерии, по которым этот выбор нужно делать осознанно, а не случайно.
**Критерий 1. Доступность без банковской карты или с минимальным бесплатным лимитом.**Некоторые провайдеры требуют привязки платёжного инструмента даже для получения бесплатного лимита. Если это создаёт проблему — ищите провайдера, предоставляющего тестовый или учебный доступ без привязки карты.
**Критерий 2. Совместимость с форматом OpenAI ****chat**** ****completions****.**Большинство современных провайдеров предоставляют именно такой интерфейс — убедитесь, что выбранный провайдер не является исключением, иначе обёртка llm_client.py, которую вы создадите на ЛР 1.2, потребует специфической для одного провайдера реализации.
**Критерий 3. Возможность работы в учебном контексте.**Некоторые корпоративные API накладывают ограничения на типы задач (например, запрет на обработку кода) или требуют дополнительного согласования для учебного использования. Убедитесь, что выбранный провайдер позволяет использовать API для задач, которые вы будете решать в курсе.
**Критерий 4. Наличие механизма отслеживания расхода токенов в ответе API.**Обязательное требование инфраструктурного документа (раздел 4.6) — логировать расход токенов при каждом вызове. Убедитесь, что API провайдера возвращает информацию о числе использованных токенов в каждом ответе (поле usage в стандартном OpenAI-формате).
**Критерий 5. Возможность локального развёртывания как резерв.**Согласно методическим указаниям к ЛР 2.2 (раздел 3.2), для атак на агента рекомендуется использовать локально развёрнутую модель, а не внешний API. Изучите возможность запуска небольшой модели локально (например, через инструменты класса Ollama) уже сейчас — это позволит на ЛР 2.2 использовать изолированную среду без специальной подготовки в момент занятия.
**6.3. Пошаговая инструкция: получение API-ключа**
Конкретная последовательность действий зависит от выбранного провайдера, поэтому ниже описана **общая процедура**, применимая к большинству провайдеров:
**Шаг 6.3.1.** Перейдите на сайт провайдера и создайте аккаунт. Используйте корпоративный или учебный email, если провайдер даёт на это скидку или расширенный лимит.
**Шаг 6.3.2.** Пройдите процедуру верификации (если требуется). Верификация может включать подтверждение email, телефонного номера, или (в редких случаях) проверку документов. Именно этот шаг может занять от минут до дней.
**Шаг 6.3.3.** Перейдите в раздел API (часто называется «API keys», «Developers», «Dashboard» или аналогично). Создайте новый API-ключ.
**Шаг 6.3.4.** Скопируйте ключ и **немедленно сохраните его в безопасном месте** — большинство провайдеров показывают ключ только один раз при создании. Если потеряете — придётся создавать новый.
**Шаг 6.3.5.** Ознакомьтесь с информацией о бесплатном лимите: сколько токенов доступно, есть ли ограничения по времени (например, «5000 токенов в день»), есть ли ограничения по типам запросов.
**6.4. Правильное хранение API-ключа: ****файл **.env
API-ключ является секретом — его нельзя хранить в коде, нельзя коммитить в репозиторий (даже в частный), нельзя передавать через мессенджеры в открытом виде. Это не параноя — это стандартная инженерная практика, и нарушение её может привести к нежелательным финансовым последствиям (ключ, попавший в публичный репозиторий, может быть использован чужими автоматическими ботами за ваш счёт в течение минут).
**Стандартный способ безопасного хранения — ****файл **.env**:**
Создайте в корне директории проекта файл .env:

API_KEY=ваш_реальный_ключ_здесь
API_BASE_URL=https://api.провайдер.com/v1  # если отличается от стандартного
MODEL_NAME=название-модели-провайдера
**Убедитесь, ****что **.env** добавлен ****в **.gitignore (это должно было быть сделано на шаге 5.3.6). Проверьте явно:
cat .gitignore | grep .env
Вы должны увидеть строку .env в выводе. Если её нет — добавьте.
**Как читать переменные ****из **.env** в Python:**

**Создайте ****файл **.env.example — шаблон без реальных значений, который можно и нужно коммитить в репозиторий:

**7. Задача 3: Изучение документации выбранных инструментов**
**7.1. Зачем изучать документацию — по существу, а не формально**
В информатике бытует полушутливый принцип RTFM («Read The Fine Manual» — прочитай документацию). Но за этим принципом скрыта серьёзная инженерная реальность: инструмент, документация которого не прочитана, используется случайным, а не осознанным образом. Вы нажимаете кнопки и смотрите, что получилось, вместо того чтобы понимать, что должно получиться и почему.
Для этого курса это особенно важно, потому что на занятиях вы не будете изучать конкретные инструменты — вы будете изучать принципы и паттерны. Конкретный инструмент вы выбрали сами; значит, и разобраться с ним — ваша ответственность. Аудиторное время не предназначено для объяснения «как запустить именно этот агент на вашей конкретной системе».
**7.2. Что именно нужно изучить в документации CLI-агента**
Ниже перечислены конкретные вопросы, на которые вы должны знать ответ после изучения документации выбранного CLI-агента. Это не «прочитать всё подряд», а целенаправленный поиск информации:
**Вопрос 1. Как запустить агента, указав конкретный файл или директорию для работы?**Найдите в документации: есть ли параметр командной строки, ограничивающий область видимости агента? Это важно для ЛР 2.2 (атаки), где вы будете хотеть ограничить доступ агента к определённым директориям.
**Вопрос 2. Как задать системный ****промпт**** агенту — то есть начальную инструкцию, определяющую его роль и поведение?**Это центральный механизм, обсуждаемый на лекции 1.1. Поймите, как именно ваш конкретный инструмент позволяет задавать системный промпт: через аргумент командной строки, через конфигурационный файл, или каким-то ещё способом.
**Вопрос 3. Как работает режим «автоматического подтверждения» — выполняет ли агент изменения немедленно или спрашивает разрешения?**Разные инструменты ведут себя по-разному: одни применяют изменения сразу (что быстро, но рискованно), другие показывают предложенные изменения и ждут явного подтверждения (медленнее, но безопаснее). Поймите, как настроить нужный режим.
**Вопрос 4. Как агент работает с историей изменений — создаёт ли он коммиты в ****git**** автоматически?**Это прямо релевантно для операции 5 на ЛР 1.1.
**Вопрос 5. Есть ли режим «только чтения» — когда агент может анализировать код, но не может его изменять?**Это важно как для безопасности работы (на ранних этапах обучения), так и для понимания, как ограничивать права агента.
**7.3. Что именно нужно изучить в документации API провайдера модели**
**Вопрос 1. Какова структура запроса и ответа?**Вы уже видели базовый пример в разделе 6.5. Теперь поймите детали: что находится в поле messages, какие роли поддерживаются (system, user, assistant), что значат поля temperature и max_tokens.
**Вопрос 2. Как посмотреть историю использования и текущий остаток лимита?**Большинство провайдеров предоставляют либо страницу в личном кабинете, либо API-эндпоинт для проверки баланса и истории. Найдите этот способ — он понадобится для мониторинга расхода на протяжении всего курса.
**Вопрос 3. Как рассчитать стоимость запроса?**Если провайдер платный — найдите страницу с тарифами (pricing). Если используете бесплатный лимит — узнайте, как именно токены считаются и что происходит при исчерпании лимита (блокировка всех запросов или только превышающих лимит).
**Вопрос 4. Какие параметры влияют на «случайность» ответа модели?**Параметр temperature (от 0 до 1 или 2 в зависимости от провайдера) — базовый параметр, влияющий на детерминированность ответа. Значение 0 — максимально детерминированный ответ; высокое значение — более «случайный», разнообразный. Поймите, как менять этот параметр и зачем.

**9. Типичные ошибки выполнения (развёрнуто)**
**Ошибка 1. Установка окружения без тестового вызова — «должно работать», но не проверено.***Как это выглядит на практике:* все пакеты установлены, CLI-агент установлен, но вы ни разу не запустили реальный вызов к API, предполагая, что «если установилось — значит, работает». *Почему это происходит:* установка пакета и работоспособность системы — разные вещи; между ними стоят ещё корректный API-ключ, верный URL, подходящее название модели. *Чем это оборачивается:* первое занятие ЛР 1.1 начинается с отладки тестового вызова вместо исследования поведения агента. *Как избежать:* обязательно выполните скрипт из раздела 6.5 и убедитесь в получении содержательного ответа.
**Ошибка 2. Слепое копирование окружения товарища по команде.***Как это выглядит на практике:* вместо самостоятельной установки и проверки, участник команды берёт готовую папку с окружением у товарища или просит его «удалённо всё сделать». *Почему это происходит:* кажется быстрее и надёжнее, особенно если у товарища «уже всё работает». *Чем это оборачивается:* участник не знает, из чего состоит его окружение, не может самостоятельно отладить проблему, и на экзамене не способен объяснить, как воспроизвести систему. *Как избежать:* помощь товарища допустима (объяснить, подсказать команду, помочь разобраться с ошибкой), но каждый участник лично выполняет каждый шаг на своей машине.
**Ошибка 3. Откладывание получения API-ключа на последний день.***Как это выглядит на практике:* участник откладывает регистрацию у провайдера до вечера перед ЛР 1.1, обнаруживает, что верификация занимает 24–48 часов, и приходит на занятие без работающего доступа. *Почему это происходит:* кажется, что «это займёт 5 минут, успею в последний момент». *Чем это оборачивается:* занятие пропущено содержательно. *Как избежать:* выполнить шаги 6.3.1–6.3.3 в первый же день после получения этого задания.
**Ошибка 4. Изучение документации «по диагонали» без пробного запуска.***Как это выглядит на практике:* участник пробегает глазами документацию, отмечает «ознакомился», но не пробует ни одной команды из неё самостоятельно. *Почему это **происходит:* чтение воспринимается как «изучение», хотя понимание без практики не формируется. *Чем это оборачивается:* на ЛР 1.1, при первом реальном использовании инструмента, возникают простейшие вопросы («а как передать файл агенту?»), которые должны были быть выяснены заранее. *Как избежать:* для каждого из пяти вопросов раздела 7.2 не просто найдите ответ в документации, но и выполните соответствующую команду практически — убедитесь, что она работает как описано.

**11. Частые вопросы**
**Что делать, если выбранный провайдер требует банковскую карту, а у меня её нет?**Это именно тот сценарий, о котором предупреждает раздел 6.2, критерий 1. Выберите другого провайдера, предоставляющего доступ без привязки карты, или рассмотрите вариант с локально развёрнутой моделью — для большинства задач курса (кроме самых сложных) относительно небольшая локальная модель вполне справляется.
**Нужно ли устанавливать CLI-агент именно в виртуальное окружение, или он может быть установлен глобально?**Предпочтительно — в виртуальное окружение, чтобы версия инструмента была зафиксирована в requirements.txt и воспроизводима. Если инструмент устанавливается только глобально (например, через самоустановщик, а не через pip) — зафиксируйте версию вручную в отчёте о готовности и в будущем stack.md.
**Что если я уже пользовался одним из инструментов раньше в другом курсе — нужно ли создавать новое окружение?**Да, нужно. Другой курс — другой проект, другой набор зависимостей, другая версия инструмента. Даже если вы раньше использовали аналогичный инструмент — создайте чистое, новое, изолированное окружение специально для этого курса.
**Как узнать, сколько токенов примерно будет стоить одна лабораторная работа, чтобы правильно оценить бесплатный лимит?**Ориентировочный расчёт: запрос к агенту на анализ файла из 100 строк кода обычно использует 500–1500 входящих токенов (в зависимости от длины системного промпта и самого файла) и 200–800 исходящих. Одна операция в ходе лабораторной работы — примерно 1000–2000 токенов суммарно. За одно занятие с пятью операциями — порядка 5000–20000 токенов. Если бесплатный лимит провайдера составляет, например, 1 миллион токенов — его должно хватить на весь курс с запасом. Если лимит существенно меньше — учтите это при выборе провайдера.
